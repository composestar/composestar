Build-in Filter Actions

The following build-in filter actions are available:

Filter Name Description
AdviceAction
Calls the substitution-message with the JoinPointContext as an argument and then continues in the filterset. Can be used for example to implement certain advicebehaviour before or after a dispatch.
ContinueAction
Does nothing.
DispatchAction
Dispatches to the substitution-message and returns afterwards.
ErrorAction
Throws an exception.
SubstitutionAction
Has only filterbehaviour (empty execute method); substitutes the message with the substitutionmessage and continues.

Defining new filter actions

A new filter action can be defined by creating a class extending the abstract class FilterAction . This enforces the implementation of an Execute method, which should implement the behaviour of the filteraction.
This method is called when the filter action is encountered in a filterset and is provided with the JoinPointContext object. The filter action class should also contain a custom attribute FilterActionAttribute , containing the following information needed by the compiletime:

  • The unique name of the filter action.
  • The Flow behaviour in the filterset.
  • The Message-substitution behaviour: Whether the message is still the original message, the substituted message or any message.
  • Whether the JoinPointContext needs to be supplied (optional). The default value is true, but disabling this can give a slight increase in performance, at the cost of not having the JPC information available in the execute method.
